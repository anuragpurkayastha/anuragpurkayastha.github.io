      Designing An App - Part 1

      October 11, 2021 5 minute read

      Recently I had to complete a week long coding test for a job application I submitted. Fortunately, the company found my submission of acceptable quality and I made it through to the next round.

      This post is about the design process I undertook to program the application.

      The Game

      The game was a customised version of the popular game, Monopoly. The rules of the game are as follows.

      There are four players who take turns in the following order:

      1. Peter
      2. Billy
      3. Charlotte
      4. Sweedal
      • Each player starts with $16.
      • Everybody starts on GO
      • You get $1 when you pass GO (this excludes your starting move)
      • If you land on a property, you must buy it
      • If you land on an owned property, you must pay rent to the owner
      • If the same owner owns all property of the same colour, the rent is doubled
      • Once someone is bankrupt, whoever has the most money remaining is the winner
      • There are no chance cards, jail or stations
      • The board wraps around (i.e. you get to the last space, the next space is the first space)

      The squares of the board were provided as a JSON file named board.json. This file was to be read in to the program. Another important implementation note is that the game was deterministic, meaning that there were set moves for each player which were also to be read in to the program.

      Task

      The task I was given to complete was to design a program that simulates the gameplay given the moves and board and determine the winner of each game.

      In addition the following were also required:

      • How much money does everybody finish with?
      • What locations does everybody finish on?

      Application Design

      When I first started programming in my life, I would be very inclined to start coding something straight away without any planning (or more accurately, I would plan as I was writing the code).

      However, with more experience I have come to understand that planning the approach before typing any code makes things a whole lot easier. So, this is exactly what I did to tackle this test.

      Object Oriented Progamming

      This application was a perfect opportunity for object oriented programming because there are different objects that could be implemented.

      Thinking about the game, what objects can there be? Well, there are Players who earn money from rent, spend money to buy property or paying rent. There are also Square objects that represent each square on the board (GO, property etc). Finally, a Game object can be used to create a game of Players and Squares.

      Summarising, the main objects are as follows:

      1. Players
      2. Squares (of the board)
      3. Game - the main object which creates a game with a board of squares and players.

      Players

      In a game of monopoly, each player has a name and money that they spend on buying property and paying rent. The amount of money also determines the winner of the game. For the purpose of designing a simulation, a Player can also keep track of its own position on the board.

      So, the attributes that a Player of the game has are:

      1. Name
      2. Total amount of money
      3. Current position on the board

      These can be the fields of a Player object.

      What can a player do? Well, they can:

      1. Spend money - for rent or buying property
      2. Earn money - from rent or passing GO
      3. Check if they are bankrupt - in which case the game ends if any player goes bankrupt

      These can be the methods of a Player object.

      Squares

      A real monopoly game board is made up of board Squares, with each square containing a property (that can be bought or rented) or GO. So, extracting from reality, we can design a board of the simulation to be made up of Square objects with similar properties. That is, each Square can be a property that can be bought (owned) and rented.

      So each Square has the following attributes:

      1. Name (Property name)
      2. Price
      3. Colour
      4. Type (“property” or “go”)
      5. Owner (when bought)
      6. Rent (when bought).

      A Square has the following abilties:

      1. Being bought
      2. Double the rent (when a player owns all the properties of the same colour)
      3. Check if the property is owned.

      Game

      So now we have Players and Squares. We can implement a Game object that initialises and runs a “game” in the simulation. For example, a Game can create the squares and players and implmement the gameplay logic.

      Using these ideas, a Game object can have the following abilities:

      1. Check if any player is bankrupt
      2. Announce the winning player and the final results when game ends.
      3. Keep track of the current player.
      4. Move the current player the predetermined amount of moves.
      5. Check if any player owns all properties of one particular colour and double the rent of all corresponding properties.

      Gameplay Logic

      Now that the application objects have been designed, we can start thinking about the actual logic of the gameplay. That is, we can think about a high level idea of how a game is played. The algorithm that I used is as follows.

      1. Check if any player is bankrupt or if there are no more moves left to play. a. If any condition is met, then announce the final results.

      2. Get the current Player whose turn it is.

      3. Get the amount of moves, x, that the current Player should move.

      4. Move the Player x moves away from the Player’s current position on the board. a. If the destination square that is x moves away from the Player’s current position is beyond the total number of squares, then wrap around.

      5. Check if the square the Player has landed on is owned. a. If it is, then pay rent to the owner. b. If it is not, then Player buys the property. The owner attribute of the square will be updated to be the Player.

      6. Go to step 1.

      The steps above are essentially the steps for one turn for a Player. We can implement these steps for each Player.

      Conclusion

      In this post we have covered the design of the application by designing game objects which creates some modularity. This helps to build, debug, test and extend the application. We have also gone over the game play logic by thinking about the steps that a turn for a Player will take.

      This post is about the game simulation, but it also covers thinking about programs using Object Oriented Programming design. It shows how breaking up a seemingly complex program can simplify the implementation and maintenance of the application by breaking the program down into smaller parts.

      In Part 2, I’ll briefly go over some technical details of the implementation which will essentially be an outline of the files and unit test.
